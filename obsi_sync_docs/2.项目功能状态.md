# 项目功能状态

## 当前目标

先完成 Obsidian 配置同步 MVP：

1. 扫描 vault
2. 选择主库
3. 选择多个从库
4. 选择同步范围
5. 生成同步计划
6. 确认后执行同步
7. 输出执行结果

## 已实现功能

- M1：扫描 Obsidian 库列表，已验证。
- M2：前端主库/从库选择状态，已开发，待验证。

## 当前最小设计

| 版本 | 阶段 | 需求 | 后端接口 | 核心规则 | 状态 |
| --- | --- | --- | --- | --- | --- |
| M1 | 解析库列表 | 前端用户指定一个路径，解析 Obsidian 库并显示列表 | `ScanVaults(root string) ([]VaultInfo, error)` | `root/.obsidian/` 存在时 root 是 vault；否则最多检查到 `root/*/*/.obsidian/`；扫描子目录时跳过隐藏目录、`node_modules`、`.git` | 已验证 |
| M2 | 设置主从 | 前端用户点击一个库作为主库，多选几个库作为从库 | 无 | M2 只做前端状态管理，不新增后端接口，不做持久化 | 已开发，待验证 |
| M3 | 设置同步文件 | 展示主库里的配置文件，用户选择需要同步的文件 | 待定 | 同步路径使用相对 `.obsidian` 的路径；默认排除 `.obsidian` 根下 `workspace*` 文件 | 待开发 |
| M4 | 展示同步计划 | 执行前校验并展示复制、跳过、危险操作等计划 | `BuildSyncPlan(req SyncRequest) (SyncPlan, error)` | 只计算，不写文件；校验主库和从库路径有效；主库不能在从库列表里 | 待开发 |
| M5 | 执行同步 | 用户确认计划后复制主库文件到目标库并返回结果 | `ExecuteSyncPlan(plan SyncPlan) (SyncResult, error)` | 执行参数使用 M4 生成的计划，避免展示计划和实际执行不一致 | 待开发 |

## M1 约定

返回结果给前端展示。

```go
type VaultInfo struct {
	Path string `json:"path"`
	Name string `json:"name"`
}
```

发现某个目录已经是 vault 后，默认不继续深入该 vault 内部扫描。

M1 不做深层递归扫描。用户选择的目录应满足以下形态之一，最多识别 2 层子目录：

```text
root/.obsidian
root/<vault>/.obsidian
root/<group>/<vault>/.obsidian
```

## M2 约定

M2 只做前端选择状态，不增加后端接口。

前端状态：

```text
mainVault
targetVaults
```

用户流程：

```text
扫描 vault
展示 vault 列表
点击一个 vault 作为主库
多选若干 vault 作为从库
```

库列表持久化不属于 M2，后续作为优化项处理。

当前 UI 骨架：

当前已落地：

## M3 约定

同步文件使用相对 `.obsidian` 的路径，不使用绝对配置文件路径。

示例：

```text
appearance.json
hotkeys.json
plugins/dataview/
```

默认排除 `.obsidian` 根目录下的 `workspace*` 文件，例如：

```text
workspace.json
workspaces.json
workspace-mobile.json
```

默认规则只在用户首次进入同步范围设置时应用一次，后续允许用户编辑。

## M4 约定

`BuildSyncPlan` 只计算同步计划，不写文件，不创建备份目录。

计划需要展示：

```text
主库有哪些配置文件
哪些会复制到从库
哪些会跳过
是否存在危险操作
```

危险操作第一版只包含：镜像模式下删除从库多余配置文件。

## M5 约定

`ExecuteSyncPlan` 只接收 `SyncPlan`，不重新接收零散参数。

M5 第一版执行覆盖同步，不包含备份功能。

## 同步模式

### 覆盖模式

主库选中的配置文件覆盖到从库。

MVP 第一版只实现覆盖模式。

### 镜像模式

从库多余的配置文件也删除，使其和主库一致。

镜像模式暂不进入 MVP 第一版，后续实现前需要先完善危险操作确认。
