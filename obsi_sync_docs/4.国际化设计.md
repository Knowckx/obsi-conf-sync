# 国际化设计

> 创建日期：2026-07-23
> 最近更新：2026-07-23

## 目标

- 软件首次启动且用户没有语言设置时，跟随系统语言。
- 用户可以通过右上角语言按钮切换界面语言。
- 用户选择的语言持久化，后续启动优先使用用户设置。
- 当前支持中文和英文，结构允许后续增加其他语言。
- 切换语言时保留当前步骤、选择状态、同步计划和执行结果。
- 所有面向用户的 UI 文案均支持国际化。

## 支持语言

内部语言标识：

```ts
type Locale = 'zh' | 'en';
```

对应 HTML 语言标识：

```text
zh -> zh-CN
en -> en
```

系统语言匹配规则：

```text
navigator.languages 中首个以 zh 开头的语言 -> zh
其他语言 -> en
```

不细分简体中文和繁体中文。当前中文资源统一使用简体中文。

## 语言优先级

启动时按以下顺序确定语言：

1. 读取 `localStorage` 中的用户语言设置。
2. 没有用户设置时读取系统语言。
3. 无法识别时使用英文。

用户切换语言后：

- 立即更新全部界面文案。
- 写入 `localStorage`。
- 更新 `<html lang="zh-CN|en">`。
- 不重新加载页面，不重置业务状态。

建议使用的存储键：

```text
obsi-conf-sync.locale
```

## 前端结构

当前只有两种语言且界面规模较小，第一版使用轻量自建国际化能力，不新增第三方依赖。

建议文件：

```text
frontend/src/lib/i18n/i18n.svelte.ts
frontend/src/lib/i18n/messages.ts
```

`i18n.svelte.ts` 负责：

```ts
locale
setLocale(locale)
toggleLocale()
t(key, params?)
```

`messages.ts` 统一维护中英文翻译表：

```ts
const messages = {
  zh: {
    common: {
      next: '下一步',
    },
  },
  en: {
    common: {
      next: 'Next',
    },
  },
};
```

翻译 key 需要使用 TypeScript 类型约束，保证中英文资源结构一致，缺少翻译时在开发阶段直接暴露类型错误。

动态数字、名称和路径通过参数插值，不拼接完整句子：

```ts
t('sync.createdCount', { count })
```

英文涉及单复数时分别维护单数和复数文案，由 `count` 选择，不预先建设复杂的复数规则系统。

## 语言切换入口

在 `app.svelte` 增加应用级 Header，语言按钮位于右上角。

交互规则：

- 使用现有 `@lucide/svelte` 的 `Languages` 图标。
- 当前只有中英文，点击按钮直接切换，不增加下拉菜单。
- 按钮显示当前语言简写：中文显示 `中`，英文显示 `EN`。
- `title` 和 `aria-label` 使用当前语言。
- 支持鼠标和键盘操作。
- 窄屏下不能覆盖页面主体或造成水平滚动。

如果后续增加第三种语言，再将按钮升级为语言选择菜单。

## 文案范围

需要国际化：

- 五个同步步骤名称。
- 页面标题、说明、按钮和状态标签。
- 加载状态、空状态、确认提示和失败提示。
- 文件选择对话框标题及按钮。
- 同步计划和执行结果统计。
- 社区插件说明和配置项说明。
- `title`、`aria-label` 等辅助文案。
- 仅开发环境显示的预设入口及错误提示。
- 后端返回并最终展示给用户的业务错误。

不需要翻译：

- Vault 名称和文件系统路径。
- 配置文件及目录路径。
- 插件名称和版本。
- Obsidian、CSS 等产品名和技术名。
- 仅写入日志、不会展示到 UI 的开发信息。

## 后端动态文案

### 配置项说明

当前配置项说明由 Go 后端通过 `ConfigItem.Description` 返回中文文本。国际化后，后端不再直接决定展示语言。

建议改为：

```go
type ConfigItem struct {
    // ...
    DescriptionKey string `json:"descriptionKey"`
}
```

`conf.ConfigItemRules` 保存稳定的说明 key：

```go
"appearance.json": {
    DescriptionKey: "config.appearance",
    DefaultSelected: true,
}
```

前端根据 `DescriptionKey` 获取当前语言的说明。插件名称、版本和路径继续作为动态数据展示。

修改 Wails 后端模型后，需要执行：

```text
task bindings
```

### 后端错误

禁止根据中文错误文本进行翻译匹配。

会展示到 UI 的业务错误应提供稳定错误码，前端根据错误码选择翻译，并将路径等动态信息作为参数。

示例：

```text
VAULT_NOT_FOUND
INVALID_SYNC_PATH
MAIN_VAULT_AS_TARGET
SYNC_ALREADY_RUNNING
SYNC_ITEM_FAILED
```

未知系统错误显示本地化的通用失败提示。原始错误可作为技术详情保留，但不能代替面向用户的本地化说明。

## 实现计划

整体属于中等工作量，预计需要 14 至 22 小时。分为四个阶段实施，每个阶段完成后单独验证。

### 第一阶段：国际化基础与语言按钮

预计工作量：2 至 3 小时。

实现内容：

1. 创建语言状态、系统语言检测和类型安全翻译函数。
2. 实现用户语言设置持久化。
3. 同步更新 `<html lang>`。
4. 在应用 Header 右上角增加语言切换按钮。
5. 确认切换语言不会重置当前业务状态。

交付结果：

- 首次启动可以根据系统语言选择中文或英文。
- 用户可以手动切换语言。
- 用户选择可以在重新启动后恢复。

### 第二阶段：前端静态文案迁移

预计工作量：4 至 6 小时。

实现内容：

1. 替换所有 Svelte 页面和组件中的静态 UI 文案。
2. 国际化文件选择对话框标题及按钮。
3. 国际化 `title`、`aria-label` 等辅助功能文案。
4. 处理动态数量及英文单复数。
5. 搜索并清理遗漏的硬编码用户界面文案。

交付结果：

- 中英文界面主体均可完整使用。
- 页面、按钮、状态和空提示可以随语言立即切换。

### 第三阶段：后端动态文案改造

预计工作量：3 至 5 小时。

实现内容：

1. 将配置项模型的 `Description` 改为 `DescriptionKey`。
2. 将 `conf.ConfigItemRules` 改为维护稳定的说明 key。
3. 在前端翻译表中补充中英文配置说明。
4. 更新前端模型并执行 `task bindings`。
5. 验证 M3 配置列表、插件说明和默认选择行为。

交付结果：

- 配置项及社区插件说明使用当前界面语言。
- Go 后端不再直接返回固定中文展示文案。

### 第四阶段：错误国际化与完整验收

预计工作量：5 至 8 小时。

实现内容：

1. 为会展示到 UI 的后端业务错误增加稳定错误码。
2. 实现前端错误码翻译和未知错误兜底。
3. 分别使用中文和英文完整走通 M1 至 M5。
4. 检查英文长文案下的步骤导航、操作按钮、统计区域和窄屏布局。
5. 执行后端核心回归测试及必要的前端构建检查。
6. 更新项目功能状态文档。

交付结果：

- 已知业务错误使用当前界面语言。
- 中文和英文完整流程均通过验证。
- 最小窗口尺寸下不存在明显布局问题。

## 验收标准

- Windows 中文系统首次启动默认显示中文。
- Windows 非中文系统首次启动默认显示英文。
- 用户点击右上角语言按钮后，当前页面立即切换语言。
- 重新启动软件后保留用户最后选择的语言。
- 切换语言不改变当前步骤、Vault 选择、同步范围、同步计划和执行结果。
- 中文和英文均可完整走通 M1 至 M5。
- 文件选择对话框使用当前语言。
- 配置项说明和社区插件提示使用当前语言。
- 已知业务错误使用当前语言，不直接展示固定中文错误。
- 英文界面在最小窗口尺寸下不存在遮挡、截断和水平滚动。
- 前端不存在遗漏的硬编码用户界面文案。
