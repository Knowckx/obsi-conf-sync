# 泛用国际化模块与项目接入设计

> 创建日期：2026-07-23
> 最近更新：2026-07-23

## 目标

本次国际化分为两个交付目标：

1. 开发一个基于 Svelte 5 的泛用国际化模块。
2. 使用该模块完成 `obsi-conf-sync` 的中英文接入。

泛用模块先在当前项目中开发和验证，稳定后移动到 `infa-s5`。其他项目只需要引用模块、提供本地文字表和少量应用配置，即可获得相同的国际化能力。

## 总体架构

```text
infa-s5 / 泛用模块
        │
        │ createI18n(options)
        ▼
obsi-conf-sync / 项目实例
        ├── 本地中英文文字表
        ├── 应用存储键
        ├── HTML lang 映射
        ├── 右上角语言按钮
        └── 项目业务文案
```

模块分为三层：

```text
泛用国际化引擎
    ↓
项目国际化实例
    ↓
Svelte 页面和组件
```

## 第一部分：泛用国际化模块

### 开发位置

第一版先在当前项目开发：

```text
frontend/src/lib/i18n/
├── core/
│   ├── create_i18n.svelte.ts
│   ├── types.ts
│   └── interpolate.ts
├── messages.ts
└── index.svelte.ts
```

稳定后移动到：

```text
infa-s5/src/lib/i18n/core/
├── create_i18n.svelte.ts
├── types.ts
└── interpolate.ts
```

`core/` 是泛用模块边界：

- `create_i18n.svelte.ts`：响应式状态、语言检测、持久化、HTML lang 同步和翻译入口。
- `types.ts`：`MessageSchema`、`CreateI18nOptions`、Locale、MessageKey 和参数类型。
- `interpolate.ts`：无状态轻量参数插值。

`messages.ts` 和 `index.svelte.ts` 属于项目接入层，不进入泛用核心。当前项目不长期保留 `core/` 副本；迁移完成后统一从 `infa-s5` 导入，避免形成两套实现。

### 模块职责

泛用模块负责：

- 创建 Svelte 5 响应式语言状态。
- 从项目文字表推导语言类型和翻译 key 类型。
- 系统语言检测和支持语言匹配。
- 用户语言设置的读取与持久化。
- 当前语言切换。
- 翻译文本读取。
- 动态参数插值。
- `<html lang>` 同步能力。
- 向调用方暴露支持的语言列表。

泛用模块不负责：

- 任何 Obsidian 业务文案。
- 固定的中文或英文文字表。
- 固定的应用存储键。
- 项目 Header、页面布局和样式。
- 后端业务错误码的具体含义。
- Vault、配置项或同步流程状态。

### 工厂 API

模块对外提供：

```ts
createI18n(options)
```

`messages` 必须通过泛型参数保留调用方文字表的字面量类型，禁止在工厂入口将其收窄为普通的 `Record<string, Record<string, string>>`。

核心类型定义：

```ts
type MessageSchema = Record<string, Record<string, string>>;

type CreateI18nOptions<
  T extends MessageSchema,
  TDefaultLocale extends keyof T,
> = {
  messages: T;
  defaultLocale: TDefaultLocale;
  fallbackLocale: keyof T;
  storageKey: string;
  htmlLang?: Partial<Record<keyof T, string>>;
};

export function createI18n<
  T extends MessageSchema,
  TDefaultLocale extends keyof T,
>(
  options: CreateI18nOptions<T, TDefaultLocale>,
)
```

`defaultLocale` 是类型设计中的翻译基准语言，`MessageKey` 只从该语言的文字表推导。`fallbackLocale` 是运行时概念，仅决定当前语言缺少文案时使用哪种语言，不能参与 `MessageKey` 的生成，也不能被 `defaultLocale` 替代。

```ts
type MessageKey = Extract<
  keyof T[TDefaultLocale],
  string
>;
```

项目调用示例：

```ts
export const i18n = createI18n({
  messages,
  defaultLocale: 'en',
  fallbackLocale: 'en',
  storageKey: 'obsi-conf-sync.locale',
  htmlLang: {
    zh: 'zh-CN',
    en: 'en',
  },
});
```

泛型 `T` 必须直接从项目传入的 `messages` 推导。不能先将项目文字表声明为 `MessageSchema`，否则语言 key 和翻译 key 会被拓宽为 `string`。

```ts
const messages = {
  zh: {
    'common.next': '下一步',
  },
  en: {
    'common.next': 'Next',
  },
};

const i18n = createI18n({
  messages,
  defaultLocale: 'en',
  fallbackLocale: 'en',
  storageKey: 'obsi-conf-sync.locale',
});

i18n.t('common.next'); // 通过
i18n.t('common.missing'); // TypeScript 报错
```

这是泛用模块最核心的类型契约：项目增加、删除或误写翻译 key 时，调用方必须获得静态类型检查，不能退化为任意字符串。

实例统一提供：

```ts
i18n.state.locale
i18n.locales
i18n.setLocale(locale)
i18n.toggleLocale()
i18n.t(key, params?)
```

`Locale` 必须从 `messages` 的语言键推导，`MessageKey` 必须从 `defaultLocale` 对应文字表的 key 推导，不在泛用模块中写死：

```ts
type Locale = 'zh' | 'en';
type MessageKey = 'common.next' | 'common.previous' | ...;
```

### 响应式状态

使用 Svelte 5 `.svelte.ts` 通用响应式模块：

```ts
const state = $state({
  locale: initialLocale,
});
```

对外暴露状态对象和修改方法，不直接导出会被重新赋值的 `$state` 变量。

每个应用只创建一个国际化实例，所有页面共享该实例。切换语言只修改 `state.locale`，不重新创建应用和业务组件。

### 系统语言匹配

匹配顺序：

1. 按顺序读取 `navigator.languages`。
2. 优先匹配完整语言标识。
3. 完整标识不匹配时，使用基础语言匹配。
4. 没有匹配项时使用 `fallbackLocale`。

示例：

```text
zh-CN -> zh
zh-TW -> zh
en-US -> en
en-GB -> en
ja-JP -> fallbackLocale
```

匹配算法只依赖项目提供的语言 key，因此未来可以支持其他语言。

### 用户设置持久化

启动优先级：

1. 读取 `storageKey` 对应的用户设置。
2. 已保存的语言仍在 `messages` 中时使用该语言。
3. 没有有效用户设置时检测系统语言。
4. 无法匹配时使用 `fallbackLocale`。

调用 `setLocale()` 后：

- 更新响应式语言状态。
- 写入项目提供的 `storageKey`。
- 更新 `<html lang>`。

模块不能使用固定存储键，避免多个应用互相污染。

`<html lang>` 必须由泛用模块自动维护，并覆盖两个时机：

1. `createI18n()` 完成初始语言检测后立即同步。
2. 每次调用 `setLocale()` 后同步。

```ts
function syncHtmlLang(locale: Locale): void {
  if (typeof document === 'undefined') {
    return;
  }
  document.documentElement.lang =
    options.htmlLang?.[locale] ?? String(locale);
}

const state = $state({
  locale: getInitialLocale(),
});

syncHtmlLang(state.locale);

function setLocale(locale: Locale): void {
  state.locale = locale;
  if (typeof localStorage !== 'undefined') {
    localStorage.setItem(options.storageKey, String(locale));
  }
  syncHtmlLang(locale);
}
```

初始化同步不能依赖用户先切换一次语言。`htmlLang` 未提供当前语言映射时，直接使用内部语言标识；`document` 不存在时跳过 DOM 更新。

翻译文本的运行时查找顺序：

1. 当前语言。
2. `fallbackLocale`。
3. 原始翻译 key。

`defaultLocale` 和 `fallbackLocale` 当前都使用 `en`，但两者必须保持独立。未来可在不改变 `MessageKey` 推导规则的情况下调整运行时回退语言。

`t()` 禁止返回 `undefined`、空字符串或因缺少模板而抛出异常。当前语言和 `fallbackLocale` 都没有对应文案时，必须原样返回传入的 key：

```ts
i18n.t('sync.xxx'); // "sync.xxx"
```

返回 key 可以让缺失翻译在 UI 和人工验证中直接暴露，避免界面出现无提示空白。

### 翻译表约束

文字表使用扁平 key：

```ts
export const messages = {
  zh: {
    'common.next': '下一步',
  },
  en: {
    'common.next': 'Next',
  },
};
```

要求：

- `defaultLocale` 对应文字表是完整的翻译基准，以它的 key 生成 `MessageKey`。
- 其他语言允许暂时缺少翻译，运行时按回退顺序读取。
- 基准语言缺少翻译或调用不存在的 key 时，在 TypeScript 检查阶段暴露。
- 运行时意外缺少当前语言和 fallback 文案时原样返回 key。
- 动态内容通过参数插值，不拼接完整句子。

示例：

```ts
i18n.t('sync.createdCount', { count: 3 });
```

第一版只提供轻量参数插值：

```ts
type TranslationParams = Record<string, string | number>;

function interpolate(
  template: string,
  params: TranslationParams,
): string {
  return template.replace(/\{(\w+)\}/g, (placeholder, name: string) => {
    const value = params[name];
    return value === undefined ? placeholder : String(value);
  });
}
```

```text
模板：Created {count} files
参数：{ count: 3 }
结果：Created 3 files
```

插值规则：

- 参数值只支持 `string` 和 `number`。
- 同一参数可以在模板中出现多次，全部替换。
- 模板中的参数缺失时保留原占位符，避免静默产生残缺句子。
- 传入但未在模板中使用的参数直接忽略。
- 不处理复数、日期、数字格式化、选择分支或嵌套表达式。

英文单复数由项目文字表维护单数和复数 key，不引入 ICU MessageFormat。

## 第二部分：本项目接入

### 项目文件

```text
frontend/src/lib/i18n/
├── core/
│   ├── create_i18n.svelte.ts
│   ├── types.ts
│   └── interpolate.ts
├── messages.ts
└── index.svelte.ts
```

`core/` 负责泛用国际化能力，稳定后整体迁移到 `infa-s5/src/lib/i18n/core/`。

`messages.ts` 只负责：

- 中文文字表。
- 英文文字表。
- 本项目翻译 key。

`index.svelte.ts` 只负责：

- 引用 `createI18n()`。
- 传入本项目文字表。
- 设置 default 语言。
- 设置 fallback 语言。
- 设置存储键。
- 设置 HTML 语言映射。
- 导出本项目唯一的 i18n 实例。

页面只从项目实例导入，不直接创建新的国际化状态：

```ts
import { i18n } from '@/lib/i18n';
```

### 本项目语言规则

当前支持：

```text
zh -> 简体中文
en -> 英文
```

对应 HTML 标识：

```text
zh -> zh-CN
en -> en
```

当前不区分简体中文和繁体中文，所有 `zh-*` 系统语言统一匹配到 `zh`。

本项目存储键：

```text
obsi-conf-sync.locale
```

### 语言切换入口

在 `app.svelte` 增加应用级 Header，语言按钮放在右上角。

交互规则：

- 使用现有 `@lucide/svelte` 的 `Languages` 图标。
- 当前只有两种语言，点击后直接切换，不增加下拉菜单。
- 显示当前语言简写：中文显示 `中`，英文显示 `EN`。
- `title` 和 `aria-label` 使用当前语言。
- 支持鼠标和键盘操作。
- 切换语言不重置当前步骤和任何业务状态。
- 窄屏下不能覆盖页面主体或造成水平滚动。

语言按钮属于项目 UI，第一版不移动到 `infa-s5`。后续确认多个项目使用相同交互后，再单独评估泛用语言选择组件。

### 文案范围

需要国际化：

- 五个同步步骤名称。
- 页面标题、说明、按钮和状态标签。
- 加载状态、空状态、确认提示和失败提示。
- 文件选择对话框标题及按钮。
- 同步计划和执行结果统计。
- 社区插件说明和配置项说明。
- `title`、`aria-label` 等辅助文案。
- 仅开发环境显示的预设入口及错误提示。
- 后端返回并最终展示给用户的业务错误。

不需要翻译：

- Vault 名称和文件系统路径。
- 配置文件及目录路径。
- 插件名称和版本。
- Obsidian、CSS 等产品名和技术名。
- 仅写入日志、不会展示到 UI 的开发信息。

### 后端配置说明

当前配置项说明由 Go 后端通过 `ConfigItem.Description` 返回固定中文文本。国际化后，前端展示改用稳定的 `DescriptionKey`，同时保留不参与序列化的 `Description` 作为 Go 内部 fallback。

建议改为：

```go
type ConfigItem struct {
    // ...
    DescriptionKey string `json:"descriptionKey"`
    Description    string `json:"-"`
}
```

两个字段的职责必须分开：

- `DescriptionKey` 是前端展示契约，通过 Wails bindings 传给前端。
- `Description` 仅供 Go 内部 fallback、诊断或日志使用。
- `Description` 使用 `json:"-"`，禁止作为固定中文文案传给前端。
- 前端找不到 `DescriptionKey` 对应翻译时，遵循 `t()` 的缺失翻译策略并显示 key，不读取 `Description`。

`conf.ConfigItemRules` 同时保存稳定说明 key 和后端 fallback：

```go
"appearance.json": {
    DescriptionKey: "config.appearance",
    Description: "外观、主题和 CSS 片段启用状态",
    DefaultSelected: true,
}
```

前端根据 `DescriptionKey` 从本地文字表读取说明。插件名称、版本和路径继续作为动态数据展示。

修改 Wails 后端模型后执行：

```text
task bindings
```

### 后端错误

禁止根据中文错误文本进行翻译匹配。

会展示到 UI 的业务错误提供稳定错误码，前端根据错误码选择本地翻译，并将路径等动态信息作为参数。

示例：

```text
VAULT_NOT_FOUND
INVALID_SYNC_PATH
MAIN_VAULT_AS_TARGET
SYNC_ALREADY_RUNNING
SYNC_ITEM_FAILED
```

未知系统错误显示本地化通用提示。原始错误可作为技术详情保留，但不能代替面向用户的本地化说明。

## 验收标准

### 泛用模块

- 不写死 `zh`、`en` 或任何项目文案。
- `CreateI18nOptions<T, TDefaultLocale>` 保留调用方文字表的字面量类型。
- Locale 从 `T` 推导，MessageKey 只从 `T[TDefaultLocale]` 推导，且不退化为 `string`。
- `i18n.t()` 传入不存在的翻译 key 时产生 TypeScript 错误。
- `defaultLocale`、`fallbackLocale` 和 `htmlLang` 的语言键受 `keyof T` 约束。
- 当前语言缺少文案时回退到 `fallbackLocale`。
- 当前语言和 fallback 都缺少文案时原样返回 key。
- `t()` 不因缺失翻译返回 `undefined`、空字符串或抛出异常。
- `t(key, params)` 能替换字符串和数字参数，以及同一参数的多次出现。
- 缺少插值参数时保留原占位符，多余参数不影响输出。
- 泛用模块不引入 ICU MessageFormat。
- `createI18n()` 初始化后立即同步 `<html lang>`，无需用户先切换语言。
- `setLocale()` 每次切换语言后同步 `<html lang>`。
- `htmlLang` 缺少映射时使用内部语言标识，无 `document` 环境不报错。
- 支持系统语言匹配、fallback 和用户设置持久化。
- 存储键和 HTML lang 映射由项目配置。
- 多次调用 `t()` 不创建额外状态。
- 泛用代码全部位于 `i18n/core/`，项目文字表和实例配置位于其外部。
- `core/` 移动到 `infa-s5` 后，项目只需要修改导入路径。

### 本项目

- Windows 中文系统首次启动默认显示中文。
- Windows 非中文系统首次启动默认显示英文。
- 用户点击右上角语言按钮后立即切换语言。
- 重新启动软件后保留用户最后选择的语言。
- 切换语言不改变当前步骤、Vault 选择、同步范围、同步计划和执行结果。
- 中文和英文均可完整走通 M1 至 M5。
- 文件选择对话框使用当前语言。
- 配置项说明和社区插件提示使用当前语言。
- Wails bindings 包含 `descriptionKey`，不包含后端内部 `description` fallback。
- 已知业务错误使用当前语言。
- 英文界面在最小窗口尺寸下不存在遮挡、截断和水平滚动。
- 前端不存在遗漏的硬编码用户界面文案。
