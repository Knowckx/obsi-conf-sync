# Wails WebView2 视口缩放问题

> 记录时间：2026-07-16

## 现象

Windows 150% 显示缩放下，Wails 页面内容视觉上明显偏小，`max-w-6xl` 页面主体只占窗口约三分之一。

## 当前观测

- Wails：`v3.0.0-alpha2.104`
- WebView2 封装：`v1.0.24`
- WebView2 Runtime：`150.0.4078.65`
- 当前屏幕：逻辑尺寸 `1707×1067`，物理尺寸 `2560×1600`，`ScaleFactor=1.5`
- Wails 窗口尺寸：约 `1180×838`
- Wails `GetZoom()`：`1`
- DOM：`devicePixelRatio=0.5`
- DOM 视口：约 `3498×2404`
- `html/body` CSS `zoom=1`，无 `transform`
- WebView2 渲染进程参数包含 `--device-scale-factor=1.5`

正常情况下，约 1180 宽的 Wails 窗口不应得到 3498 宽的 CSS 视口。

## `max-w-6xl` 不是根因

`frontend/src/lib/TempComps/page.svelte` 当前样式：

```css
.page {
  width: 100%;
  max-width: 72rem;
  margin-inline: auto;
}
```

这组约束有价值，应保留：内容尽量占满，但最大不超过 1152px，并在宽屏居中。

当前 `main` 计算宽度为 1152px，是因为异常 DOM 视口宽达 3498px。若视口正常接近 1180px，1152px 会基本占满窗口。删除 `max-width` 只能掩盖视口换算问题。

## 根因线索

Wails 的 WebView2 封装在控制器创建时：

1. 使用 raw-pixels bounds 模式；
2. 将 `ShouldDetectMonitorScaleChanges` 设为 `false`；
3. 没有在控制器首次创建后立即设置 `RasterizationScale`。

微软约定：关闭自动检测后，宿主应用必须自行维护 `RasterizationScale`。

Wails 已有同类问题：

- Issue：<https://github.com/wailsapp/wails/issues/5544>
- 修复 PR：<https://github.com/wailsapp/wails/pull/5572>

现有修复只在最小化恢复和 `WM_DPICHANGED` 时重同步。当前问题疑似是同一根因的启动阶段变体。

## 下一步

1. 最小化并恢复窗口，重新读取 `devicePixelRatio`、`innerWidth` 和截图，确认已有重同步逻辑是否修复页面。
2. 若恢复后正常，在 Wails `setupChromium()` 的 `chromium.Embed(w.hwnd)` 后调用：

```go
w.resyncWebviewRasterizationScale()
```

后续已有的 `chromium.Resize()` 会按新比例重新布局。
3. 通过本地 Wails 分支和 `go.mod replace` 验证，不直接修改 Go 模块缓存。
4. 验证通过后向 Wails 上游反馈启动阶段场景。

## 禁止采用的补偿方案

- 删除 `mx-auto` / `max-w-6xl`
- CSS `zoom`
- 将 Wails `Zoom` 硬编码为 `3`
- 硬编码 `--force-device-scale-factor=1.5`

这些方案会在不同 DPI 或显示器环境下再次失效。
