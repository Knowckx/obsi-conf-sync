
## GO 语言约定
> 文档更新时间 2026-07-15

#### 错误处理约定
默认使用 `github.com/cockroachdb/errors`
- 调用标准库或第三方库返回错误时，假如有明确且有价值的调用上下文，优先使用 `errors.Wrap`；否则使用 `errors.WithStack`
- 错误进入项目错误链后，后续层级禁止重复 `WithStack`；只有需要补充本层语义时才使用 `errors.Wrap`。
- 错误日志统一由入口记录，传递过程中禁止重复打印。

#### 测试文件约定
项目中的测试默认定位为“开发观察型测试”，用于开发过程中快速调用函数并打印结果，方便人工判断结果是否符合预期，不默认作为自动化回归测试。

- 测试文件统一使用 `api_test.go`。
- 优先使用独立包名 `package {package_name}_test`
- 测试函数命名为 `func Test_Xxxxx(t *testing.T)`。
- 只调用目标函数的最小成功路径。
- error 使用 `ops.MustNoErrInTest(t, err)`。
- 返回结果默认使用 `log.Printf` 打印，交由开发者人工观察。
- 除非用户明确要求验证结果，否则不要测试私有实现细节、扩展边界失败用例、构建完整测试体系

#### 包级全局风格约定
对于进程内理论上唯一、无需多实例、无需外部注入的模块，默认使用“包级门面 + 包内单例实现”风格。

``` go
// 示例
var userMgr = newUserManager()

type userManager struct {
    // internal state
}
```
约束:
- 单例实例使用小写变量，仅包内可见。
- 外部通过包级函数访问，不直接暴露实现对象。
- 内部状态优先归属于单例结构体，不拆散为多个无归属包级变量。
- 无状态工具函数无需为了统一风格强行引入单例。
- 需要依赖替换、测试注入、多实例管理的模块不要使用此模式。

#### 类型风格
- 包对外暴露的 API 契约，参数和返回值优先使用项目级通用类型。
- 包内部负责完成：项目级通用类型 和 底层实现类型（SDK、第三方库、HTTP接口的类型系统等）之间的转换。

约束：
不向外暴露底层实现类型。
业务含义明确且跨模块传递的数据优先定义项目类型。
基础类型和标准库通用类型无需强制包装。


### 其他细节
- 未经允许，禁止执行 go build
- 有限取值需求，优先类型安全枚举
- Go 方法接收者统一使用 `t`
