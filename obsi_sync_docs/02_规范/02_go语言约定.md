
## GO 语言约定
> 文档更新时间 2026-07-12

#### 错误处理约定
默认使用 `github.com/cockroachdb/errors`
- 底层函数只负责创建或者返回错误，
- 中间层向上传递，需要增加环境信息使用 errors.Wrap。
- 入口处统一处理错误
- 错误传递链中禁止重复打印日志

#### 项目内测试文件约定
- 当用户要求创建测试文件，并且此目录内存在稳定的对外契约，创建 `api_test.go`
- 使用独立包名 `package {package_name}_test`
- 测试目标是验证公开 API 的最小正确使用流程：
按外部调用者方式调用公开 API
只保留最小成功路径。不测试私有实现细节、不主动扩展边界失败用例、不构建完整测试体系
- 测试函数命名统一采用项目约定：`func Test_Xxxxx(t *testing.T)`。 不使用 Go 官方推荐的 TestXxxxx 命名风格。
- 检查error 用 `ops.MustNoErrInTest(t, err)`


#### 包级全局风格约定
对于进程内理论上唯一、无需多实例、无需外部注入的模块，默认使用“包级门面 + 包内单例实现”风格。

``` go
// 示例
var userMgr = newUserManager()

type userManager struct {
    // internal state
}
```
约束:
- 单例实例使用小写变量，仅包内可见。
- 外部通过包级函数访问，不直接暴露实现对象。
- 内部状态优先归属于单例结构体，不拆散为多个无归属包级变量。
- 无状态工具函数无需为了统一风格强行引入单例。
- 需要依赖替换、测试注入、多实例管理的模块不要使用此模式。

#### 类型风格
- 包对外暴露的 API 契约，参数和返回值优先使用项目级通用类型。
- 包内部负责完成：项目级通用类型 和 底层实现类型（SDK、第三方库、HTTP接口的类型系统等）之间的转换。

约束：
不向外暴露底层实现类型。
业务含义明确且跨模块传递的数据优先定义项目类型。
基础类型和标准库通用类型无需强制包装。


### 其他细节
- 未经允许，禁止执行 go build
- 有限取值需求，优先类型安全枚举
- Go 方法接收者统一使用 `t`
