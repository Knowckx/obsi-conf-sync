## 通用开发约束
本节编码约束同时适用于各种开发语言，包含Go和前端
### 最高优先级约束 
原则01: `如无必要 勿增实体`
- 默认遵循**最佳实践+轻量**的风格去写代码
- 新功能第一版落地代码，永远遵循`只保留最小必要实现`。
- 用户的注释属于用户资产，未经允许不能擅自删除修改，除非注释明确错误
- 除非用户明确要求，否则禁止写这类代码: 
极端条件防御、未来扩展预埋、并发安全防御、重复调用防御、标准化校验、“也许以后有用”的状态和判断。

再次强调，因为 AI 通常会默认偏向“稳一点、通用一点、防御一点”的工程代码，这会违背这个原则，所以`原则01`被列为最高优先级

### 编码风格指导
原则02: `DRY Don't Repeat Yourself`
- 尽量复用已有的抽象，约定，逻辑，函数。
- 合理的减法优化，避免重复。

### 默认编码风格
优先遵循：`强类型的最小抽象分层风格`

### 命名规范  
- 不区分具体的语言:文件名、文件夹名统一使用下划线风格，例如 qmt_debug
- 在不损失可读性的前提下，优先使用常见短缩写，例如 `str`/`cfg`/`msg`。  
- 缩写有歧义时使用完整单词。  

### 代码注释约定
- 注释要精简，优先一行；复杂语义才允许多行。  
- 导出符号必须写注释，说明调用方需要知道的语义、约束、副作用或使用方式。
- 内部的方法/函数/模型/struct 也需要写注释
- 字段、常量、变量、短声明的简单说明，优先使用行尾注释；只有需要多行说明时，才使用前置注释。
- 不允许擅自删除修改用户的注释，除非注释明确错误。

### 文件内代码顺序约定
- 先放公开符号
- 公开符号之间也尽量按使用入口分组
- 其余按调用链顺序排布：调用者在前，被调用者紧随其后

### API设计约定
- 对外 API 按调用方视角设计，优先保证调用简单、直观、低心智负担。
- 调用方不应感知底层实现细节，复杂性应收敛在包内。
- 每个 API 语义要单一，不要耦合。
- 禁止一个 API 承载多种行为。应拆分。
- 禁止通过可选参数、nil 占位、互斥参数拼装多种行为。

### 并发安全约定
- 默认按单线程场景写代码。除非用户明确要求，或者 AI 已经从当前代码调用链中确认存在并发场景，否则禁止主动加入线程安全设计。



### 符号迁移兼容策略
- 正在工作的目录内部的引用变化：AI 直接全量修改，不保留兼容层。  
- 对外导出的符号或包路径迁移时，不要扫描全仓引用，先创建新路径下的真实实现。在旧路径保留 `Deprecated` 转发兼容层。
- 新代码统一使用新路径/新符号。
- 兼容后提示用户用 IDE 人工完成符号级迁移，最后删除兼容层。


### 清理约定
#### 低价值代码清理约定
原则: 按照“如无必要勿增实体”的风格指导，任何没有独立价值的代码都需要进行清理
举三个例子:
- 单薄 helper，合并掉。
- 低价值包装代码：只套一层、只改名、只转发等等，判断没有独立价值，默认删除。
- 冗余状态代码，凡是字段、默认实例引用、内部注入点只是把同一个对象多持有一层，没有独立价值的，默认删除

#### 微型文件清理约定：
不要保留无独立职责的单薄文件。
如果一个文件只剩几行代码，且没有作为独立文件存在的必要，默认并回最直接的使用方文件。


### 其他开发约定
- 尽量使用 guard 语句, 遵循提前返回原则, 避免使用嵌套过深的语句。
- 对启动契约做 Fail Fast，禁止静默托底
- 网络请求默认设置 3 秒超时
- 除非用户明确要求 否则不需要执行验证测试
