
在 01_开发约束 的基础上，对于go语言的开发还需要进一步叠加本文的约定。

## 项目中的开发约定

### 对外接口契约文件 `api.go` 
- 假如一个目录内是单一功能性代码，创建 `api.go` 放稳定对外接口契约。
- 默认使用最小导出面风格， 把复杂性收敛在包内。
- 对外契约的接口，其参数优先使用 项目级通用类型 ， 在包内实现 项目级通用类型 →  包内部平台类型

### 目录文档说明文档
若用户提到"需要增加目录文档"，那就在目录中增加一个说明文档，命名为 `_{功能名}.md`
文件里面有三节：快速使用示例 + 设计架构 + 目录内文件职责。
示例内容编写按：中文说明 + 调用示例 + 一个空行


### 错误处理约定
- 底层函数只负责返回错误，中间层优先向上传递错误，入口处再统一处理错误
- 默认使用包 cockroachdb/errors
    创建错误：errors.New / errors.Errorf
    中间层：errors.WithStack(err)
    最上层打印：fmt.Printf("%+v\\n", err)

### api_test.go 测试文件约定
- 若代码目录内存在 `api.go`，默认配套创建 `api_test.go` 用于对应的测试
- 直接按公开 API 的正确调用方式写
- 只保留最小成功路径
- 使用独立包名 `package {文件夹名}_test`
- 测试函数使用下划线风格 `func Test_Xxxxx(...)`
- err 用 infa.MustNoErrInTest(t, err)
- 默认在末尾使用 t.Logf("result:\n%+v", result) 打印结果，便于人工复审返回值语义。
- 默认不测试私有实现细节，不铺边界失败用例， 不扩展成完整测试体系


### 包级全局风格约定
对于进程内理论上唯一、无需多实例、无需外部注入的模块，默认使用“包级门面 + 包内单例实现”风格。

设计风格
- 对外提供包级门面。在 api.go 写外部关心的门面接口。
``` go
func GetUser(...) (...) {
    return userMgr.Build(...)
}
```
  - 包内保留真实实现结构
  - 包内声明一个小写单例实例
``` go
var userMgr = &userManager{...}
type userManager struct {
    ...
}
```
补充约束
- 外部不要暴露公开构造器，不要让调用方手动实例化该模块。
- 内部状态优先挂到这个单例实例上，不要散落成多个无归属的包级变量。
- 如果模块实际上没有内部状态，也没有独立实现价值，就直接使用包级函数，不要为了套这个模式额外创建空 struct 单例。



### 字符串键缓存
- 只要需求是“字符串键 + 并发安全 + 简单 Get/Set/Clear 缓存”，默认优先使用 `infa.StrMapCache[T]`
- 禁止在业务目录里重复手写同构的 `RWMutex + map[string]T` 小缓存，除非当前场景有 `infa.StrMapCache[T]` 明确不满足的约束
